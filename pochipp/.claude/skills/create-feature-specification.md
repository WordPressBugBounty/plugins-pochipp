# 機能仕様書作成コマンド

PMやデザイナーなどが定義する「機能要件」を元に、ユーザーとAIが協働して超詳細な仕様書を作成し、GitHub Issueとして作成するコマンドです。

## 基本方針

- **ユーザーの考えを重視**：まずユーザーが考える主要な仕様や考慮事項について十分にヒアリングします
- **AIは補完と提案を行う**：ユーザーの仕様を基に、足りない部分や詳細をAIが既存実装とドキュメントを参照して提案します
- **協働的な仕様作成**：一方的にAIが決めるのではなく、ユーザーとAIが対話しながら仕様を完成させます
- **不明点は確認する**：AIが判断に迷う場合や、重要な設計判断が必要な場合は、必ずユーザーに確認を取ります

## 実行手順

### ステップ1: ユーザーの考えを丁寧にヒアリング

まず、ユーザーが考える機能の全体像と重要な仕様について、**必要最小限かつ効果的にヒアリング**してください：

#### 必ず聞くべき質問（2つ）

1. **機能の概要と実現したいこと**
   - 「この機能の概要と、実現したい機能について、できるだけ具体的に教えてください」

2. **こだわりや制約があれば**
   - 「UI/UX、技術的な制約、セキュリティやエラーハンドリングなど、特にこだわりや気をつけたい点があれば教えてください。なければ、既存パターンに従って提案します」

#### ヒアリングの方針

- **自然に決まるものはAIが提案**: URLパス、ファイル名、技術スタック、デザインパターンなど、既存パターンから自然に決まるものは、ユーザーに質問せずにAIが提案する
- **重要な設計判断のみ質問**: ユーザーの意思決定が必要な部分（機能の目的、主要な要件、こだわりポイント）のみ質問する
- **曖昧な場合は選択肢を提示**: 複数の選択肢がある場合は、AIが推奨案を提示してユーザーに確認する

**重要**: ユーザーに過度な質問攻めをしない。本当に重要な設計判断のみを聞き、その他は既存パターンから推測してAIが提案します。

### ステップ2: 既存実装とドキュメントの分析

ユーザーから情報を得たら、**必ず以下のドキュメントと既存コードを読んで**、プロジェクトの理解を深めてください：

#### 必ず読むべきドキュメント

1. **`AGENTS.md`**: AI向けの実装ガイドを理解
2. **`docs/implementation-guide/README.md`**: 実装時の共通参照順を理解
3. **`docs/architecture-review.md`**: 構成とレビューポイントを理解
4. **`docs/coding-standards/backend-coding-standards.md`**: PHP/WordPress 側の規約を理解
5. **`docs/coding-standards/frontend-coding-standards.md`**: Block/UI 側の規約を理解
6. **`readme.txt`**: プラグイン公開情報・機能説明を確認

#### 既存コードの確認

類似機能や関連機能の既存実装を確認してください：
- PHP/WordPress 側: `pochipp.php`, `inc/`, `class/`
- Block/UI 側: `src/blocks/`, `src/js/`, `src/scss/`, `src/toolbar/`
- 必要に応じて `assets/` と `dist/` の影響も確認してください

#### 言語運用ルール

- 検討・下書きは英語ベースで整理してよい
- ドキュメントの読解・レビューコメント・最終アウトプットは日本語で統一する

### ステップ3: AIによる補完と詳細提案

ユーザーから得た情報を基に、**不足している部分や詳細をAIが提案**してください。

**提案の方針**：
- ユーザーが既に明示した仕様は、そのまま尊重して採用する
- ユーザーが言及していない部分について、既存実装のパターンを参考に提案する
- 重要な設計判断が必要な場合や、複数の選択肢がある場合は、ユーザーに確認を取る
- AIが判断に迷う場合は、迷っている点を明示してユーザーに相談する

以下の観点で、**既存実装のパターンに従って**詳細な仕様を提案してください：

#### 機能要件の提案内容（テンプレート）

以下のテンプレートに従って、機能要件を提案してください：

1. **画面・URL情報**（新規画面・導線の場合）
   - **URL/導線**: WordPress 管理画面メニュー、ブロックエディタ導線、フロント表示導線のどこに乗るかを提案
   - **画面タイトル**: ページのタイトルを提案
   - **アクセス権限**: 誰がアクセスできるか（管理者のみ、編集者以上、公開ユーザーなど）

2. **画面レイアウト・UI構成**
   - **レイアウト**: 既存の画面レイアウトパターンを参考に提案（ヘッダー、サイドバー、メインコンテンツなど）
   - **主要なUIコンポーネント**: ボタン、入力フィールド、表、カードなど、どのようなコンポーネントを配置するか
   - **配色・デザイン**: 既存のデザインシステムに従った配色を提案
   - **レスポンシブ対応**: モバイル、タブレット、デスクトップでの表示方法

3. **具体的な操作フロー**
   - 既存の類似機能を参考に、操作フローを提案
   - ユーザーがどのステップで何をクリック/入力するかを明確化
   - 画面遷移がある場合は、遷移先のURLも記載

4. **入力フィールド・フォーム**（入力がある場合）
   - **フィールド名**: 各入力フィールドの名前と目的
   - **入力タイプ**: テキスト、数値、日付、選択肢など
   - **バリデーションルール**: 文字数制限、必須項目、形式チェックなど
   - **プレースホルダー・ラベル**: 表示するテキストの提案
   - **デフォルト値**: 初期値がある場合は記載

5. **ボタン・アクション**
   - **ボタンラベル**: 各ボタンのテキスト（例: 「保存」「キャンセル」「送信」）
   - **ボタンの配置**: どこに配置するか（右下、中央など）
   - **ボタンの状態**: 有効/無効の切り替えタイミング
   - **クリック時の動作**: 何が起こるか（API呼び出し、画面遷移など）

6. **表示データ・コンテンツ**（データ表示がある場合）
   - **表示項目**: どのようなデータを表示するか
   - **表示形式**: テーブル、カード、リストなど
   - **ソート・フィルタ**: データの並び替えやフィルタリング機能の有無
   - **ページネーション**: データ量が多い場合のページ分割

7. **エラーハンドリング**
   - **バリデーションエラー**: 不正な入力時の表示方法（フィールド下に赤文字で表示、など）
   - **ネットワークエラー**: ネットワークエラー時の表示（トースト通知、モーダルなど）
   - **サーバーエラー**: サーバーエラー時の表示
   - **リトライ処理**: エラー時のリトライ処理の要否

8. **エッジケース**
   - **空データ・nullの場合**: どのように表示するか（「データがありません」など）
   - **最大値・最小値**: データの制限値とその扱い
   - **特殊文字・絵文字**: 許可するかどうか
   - **同時アクセス**: 複数ユーザーが同時操作した場合の動作

9. **UIの状態変化**
   - **ローディング状態**: データ取得中の表示（スピナー、スケルトンスクリーンなど）
   - **成功時**: 成功メッセージの表示方法（トースト通知、モーダルなど）
   - **失敗時**: エラーメッセージの表示方法
   - **無効状態**: ボタンやフィールドが無効になるタイミング

10. **データの保存・取得**
    - **保存先**: WordPress オプション、投稿メタ、ブロック属性など
    - **取得タイミング**: ページ読み込み時、ボタンクリック時など
    - **API エンドポイント**: AJAX アクションや既存API連携の利用方針

11. **権限・認証**
    - **アクセス権限**: 誰がこの機能にアクセスできるか
    - **認証要件**: 認証が必要か、特定の権限が必要か

12. **既存機能との連携**
    - **関連機能**: どの既存機能と関係があるか
    - **連携方法**: どのように連携するか
    - **影響範囲**: 既存機能への影響はあるか

#### シーケンス図の提案

mermaid形式でシーケンス図を作成してください：

- **主要な操作フロー**を視覚的に表現
- **システム間の相互作用**を明確化（ユーザー、フロントエンド、API、データベースなど）
- **既存の類似機能のシーケンス図**を参考にパターンを統一
- **エラーケース**も必要に応じて記載（opt、alt などのmermaid構文を使用）

#### 非機能要件の提案内容

既存の非機能要件を参考に、以下を提案してください：

1. **セキュリティ**: 認証・認可、データ暗号化、XSS/CSRF対策など
2. **可用性**: エラー時のフォールバック、リトライ処理、タイムアウト設定など
3. **保守性**: ログ出力、エラーメッセージの統一など
4. **実装方法や使用ライブラリ**: 使用する技術スタック、ライブラリ、フレームワーク、デザインパターン、既存コードの活用方法など
5. **監視・運用**: メトリクス収集、アラート設定など
6. **その他**: 適宜、実装内容に応じて適切に検討

具体的な数値や基準を可能な限り記載してください。
ただし、この機能のみによらない全体的なものに関しては言及不要です。

#### テストケースの提案

以下の観点からテストケースを提案してください：

1. **正常系**: 期待通りの操作で期待通りの結果が得られるケース
2. **異常系**: 不正な入力、エラー時の動作確認
3. **エッジケース**: 境界値、空データ、最大値など
4. **UI/UX**: 表示の確認、レスポンシブ対応、アクセシビリティなど
5. **セキュリティ**: 認証・認可、入力値サニタイズ、CSRF対策など

各テストケースは表形式で提案してください：
- **内容**: 何をテストするか
- **操作**: 具体的な手順
- **期待値**: 期待される結果
- **結果**: チェックボックス（`- [ ]`）

### ステップ4: 仕様の確認と対話的な修正

提案した仕様をユーザーに提示し、以下のように対話してください：

「以下の仕様を提案します。ユーザーが既に指定された部分はそのまま採用しています。
不足していた部分について、既存実装を参考に提案を追加しました。

【ユーザー指定の仕様】
- （ユーザーが明示した内容をリストアップ）

【AIが補完した仕様】
- （AIが追加提案した内容をリストアップ）

【確認が必要な点】
- （AIが判断に迷った点、複数の選択肢がある点をリストアップ）

内容を確認して、修正が必要な点や、確認事項への回答をお願いします。」

**重要**: 
- ユーザーの意図を最大限尊重し、修正依頼があった場合は迅速に対応してください
- AIが一方的に決めるのではなく、ユーザーとの対話を通じて仕様を洗練させてください
- 不明点や迷う点は、遠慮なくユーザーに確認してください

### ステップ5: 仕様書ファイルの作成

ユーザーの確認が取れたら、`.github/ISSUE_TEMPLATE/feature_specification.md`のテンプレートに従って、マークダウン形式で仕様書を生成し、`docs/tmp/issues/`配下にファイルとして保存してください。

**ファイル保存の設定**：
1. **保存先ディレクトリ**: `docs/tmp/issues/`
2. **ファイル名**: `YYYYMMDD_{issue名}.md` の形式（例: `20241118_音声録音・評価機能.md`）
3. **日付形式**: 現在の日付を使用（YYYYMMDD形式）
4. **Issue名**: 機能名をファイル名に含める
5. **雛形**: 必要に応じて `docs/tmp/issues/feature-specification-template.md` を利用する

**仕様書の内容**：

1. **概要**: 簡潔に2-3文で記載
2. **機能要件**: 箇条書きで詳細に記載。「誰が」「何を」「どのように」「いつ」を明確に
3. **シーケンス図**: mermaid形式で主要な操作フローを記載
4. **非機能要件**: 箇条書きで記載。具体的な数値や基準を可能な限り記載
5. **テストケース**: 表形式で記載。内容、操作、期待値、結果（チェックボックス）の4列
6. **補足情報**: 関連Issue、参考資料、類似機能の実装例などを記載

**ファイル作成後**：
- ファイルを作成したら、ユーザーに「`docs/tmp/issues/YYYYMMDD_{issue名}.md`に仕様書を保存しました。内容を確認して、修正が必要な箇所があれば教えてください。」と伝える
- ユーザーが修正を依頼した場合は、ファイルを直接編集して更新する
- 対話を重ねながら、ファイルを修正していく

### ステップ6: 仕様書の確定と対話

保存したファイルをベースに、ユーザーと対話を重ねて仕様書を完成させてください。

**対話の流れ**：
1. ユーザーが修正を依頼 → ファイルを編集して更新
2. 更新後、「ファイルを更新しました。他に修正が必要な箇所はありますか？」と確認
3. ユーザーが「OK」または「Issueを作成してください」と言うまで対話を継続

### ステップ7: GitHub Issueの作成

仕様書が確定したら、保存したファイルの内容をベースにGitHub Issueを作成してください。

**作成前の確認**：
- ユーザーに「この仕様でGitHub Issueを作成しますか？」と確認する
- ユーザーが承認したら、Issueを作成する

**Issue作成時の設定**：
1. **リポジトリ情報**: 現在のリポジトリのownerとrepo名を使用
2. **Issueタイトル**: `[機能仕様] {機能名}`の形式で作成
3. **Issue本文**: 保存したファイル（`docs/tmp/issues/YYYYMMDD_{issue名}.md`）の内容をそのまま使用
4. **ラベル**: `Codex Issue`を付与

GitHub Issueを作成する際は、`mcp_github_issue_write`ツールを使用してください。

**Issue作成後**：
- 作成したIssueのURLをユーザーに伝える
- 保存したファイルは、今後の参考用に残しておく（削除しない）

## 重要な注意事項

1. **ユーザーとの協働を重視**: ユーザーが考える仕様を最大限尊重し、不足部分を補完する形でAIが提案する
2. **対話的な仕様作成**: 一方的にAIが決めるのではなく、ユーザーとの対話を通じて仕様を完成させる
3. **不明点は確認する**: AIが判断に迷う場合や、重要な設計判断が必要な場合は、必ずユーザーに確認する
4. **既存パターンに従う**: 必ず既存のコードとドキュメントを確認し、同じパターンに従う
5. **具体的な提案**: 「速い」「安全」などの曖昧な表現ではなく、具体的な数値や基準を記載
6. **エッジケースを重視**: 正常系だけでなく、異常系やエッジケースも提案
7. **テンプレートに従う**: `.github/ISSUE_TEMPLATE/feature_specification.md`のテンプレート形式に必ず従う
8. **ユーザーの意図を尊重**: ユーザーが既に明示した仕様は、AIが勝手に変更しない
9. **リポジトリ実態に合わせる**: WordPress プラグイン構成（`pochipp.php`, `inc/`, `class/`, `src/`）から外れた前提を置かない
